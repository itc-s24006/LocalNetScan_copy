# ネットワーク入門 - 1問1答で学ぶネットワークの基礎

**対象者**: ネットワークの概念を初めて学ぶ学生
**目標**: ネットワークがどのように動作するか、基本的な概念を理解する

---

## 第1章: ネットワークとは何か？

### Q1: ネットワークって何ですか？

**A1**: ネットワークとは、**複数のコンピューター（PC）同士が接続され、情報をやり取りできる仕組み**のことです。

**身近な例**:
```
あなたのスマートフォン → インターネット → 友達のスマートフォン
```

この「つながり」がネットワークです。

**簡単なネットワーク図**:
```
  [PC-A] ━━━━━━━ [PC-B]

     2台のPCが直接つながっている状態
```

---

### Q2: 「つながる」とは具体的にどういうことですか？

**A2**: 「つながる」とは、**電気信号が届き、それに応答できる状態**のことです。

**3つのステップ**:
1. **電気信号を送る**: PC-AがケーブルやWi-Fiで電気信号を送信
2. **パケットとして解釈**: PC-Bがその電気信号を「パケット」（データのかたまり）として理解
3. **応答する**: PC-Bが理解した内容に対して返事（応答）を送る

**シーケンス図**:
```
   PC-A                           PC-B
    |                              |
    |--- 電気信号を送信 ----------->|
    |    (パケット: "Hello!")       |
    |                              |
    |                    [解釈] "Hello!"を受信
    |                              |
    |<-- 応答を送信 ----------------|
         (パケット: "Hi!")         |
    |                              |
  [解釈] "Hi!"を受信
    |                              |

   ✅ つながった！
```

## 第2章: 人間のコミュニケーションとの類似性

### Q3: ネットワークって人間の会話に似ていますか？

**A3**: はい、**とても似ています**！ネットワークは人間のコミュニケーションと同じ仕組みです。

**人間の会話との比較**:

| 要素 | 人間の会話 | ネットワーク |
|------|-----------|-------------|
| **伝達手段** | 空気の振動（音波） | 電気信号 |
| **言語** | 日本語、英語など | プロトコル（TCP/IP等） |
| **メッセージ** | 言葉 | パケット |
| **理解** | 相手が日本語を理解 | 相手が同じプロトコルを理解 |
| **応答** | 返事をする | ACK（確認応答）を送る |

**例: 人間の会話**:
```
田中さん: 「おはよう！」（日本語で話しかける）
         ↓
佐藤さん: （日本語を理解できる）
         ↓
佐藤さん: 「おはよう！」（日本語で返事）
         ↓
✅ コミュニケーション成功！
```

**例: ネットワーク**:
```
PC-A: パケット送信（TCP/IPプロトコルで）
      ↓
PC-B: （TCP/IPプロトコルを理解できる）
      ↓
PC-B: パケット返信（TCP/IPプロトコルで）
      ↓
✅ 通信成功！
```

**もし言語が違ったら？**:
```
田中さん: 「おはよう！」（日本語）
         ↓
Johnさん: （日本語を理解できない）
         ↓
Johnさん: 「???」
         ↓
❌ コミュニケーション失敗
```

これはネットワークでも同じです。**同じプロトコル（言語）を使わないと通信できません**。

## 第3章: パケットとは何か？

### Q4: 「パケット」って何ですか？

**A4**: パケットとは、**ネットワークで送られるデータの小さなかたまり**です。

**手紙との比較**:
- **手紙**: 封筒に入れて送る
- **パケット**: データを小さく分割して、それぞれに宛先情報を付けて送る

**パケットの構造**:
```
┌─────────────────────────────────┐
│ ヘッダー（宛先・送信元の情報）      │
├─────────────────────────────────┤
│ データ本体（実際の内容）           │
├─────────────────────────────────┤
│ フッター（エラーチェック情報）      │
└─────────────────────────────────┘
```

**例: Webページを見るとき**:
```
大きなWebページ（例: 写真入りのページ）
         ↓
  [分割して複数のパケットに]
         ↓
パケット1: [ヘッダー][データ1/10]
パケット2: [ヘッダー][データ2/10]
パケット3: [ヘッダー][データ3/10]
   ...
パケット10: [ヘッダー][データ10/10]
         ↓
  [インターネットを通って送信]
         ↓
  [受信側で組み立て直す]
         ↓
完全なWebページとして表示！
```

**なぜ分割するのか？**:
1. **効率的**: 小さく分けて複数の経路で送れる
2. **エラーに強い**: 一部が失敗しても、その部分だけ再送すればいい
3. **公平**: 大きなデータを送っている人が、ネットワークを独占しない

## 第4章: OSIモデル - ネットワークの階層構造

### Q5: ネットワークって複雑そうですが、どうやって理解すればいいですか？

**A5**: **OSIモデル**という「7階層の考え方」で理解すると分かりやすいです。

**OSI 7階層モデル**:

```
┌─────────────────────────────────────┐
│ 第7層: アプリケーション層             │ ← Webブラウザ、メール
│        (ユーザーが使うアプリ)         │
├─────────────────────────────────────┤
│ 第6層: プレゼンテーション層           │ ← データの形式変換
│        (データの表現方法)             │
├─────────────────────────────────────┤
│ 第5層: セッション層                  │ ← 通信の開始・終了
│        (通信の管理)                  │
├─────────────────────────────────────┤
│ 第4層: トランスポート層              │ ← TCP/UDP（信頼性の確保）
│        (データの信頼性)              │
├─────────────────────────────────────┤
│ 第3層: ネットワーク層                │ ← IPアドレス（経路選択）
│        (データの配送経路)            │
├─────────────────────────────────────┤
│ 第2層: データリンク層                │ ← MACアドレス（直接通信）
│        (直接つながった機器間の通信)   │
├─────────────────────────────────────┤
│ 第1層: 物理層                       │ ← ケーブル、電気信号、Wi-Fi
│        (物理的な接続)               │
└─────────────────────────────────────┘
```

**階層ごとの役割**:

| 層 | 名前 | 役割 | 具体例 |
|---|------|------|--------|
| **7** | アプリケーション層 | ユーザーが使うアプリ | Webブラウザ、メール |
| **6** | プレゼンテーション層 | データの表現形式 | 暗号化、文字コード変換 |
| **5** | セッション層 | 通信の管理 | 接続の開始・終了 |
| **4** | トランスポート層 | 信頼性の確保 | TCP（確実に届ける） |
| **3** | ネットワーク層 | 経路選択 | IPアドレス、ルーター |
| **2** | データリンク層 | 直接接続 | MACアドレス、スイッチ |
| **1** | 物理層 | 物理的接続 | ケーブル、電気信号 |

**重要なのは第1層（物理層）**:
- **すべての基礎**: 電気信号がなければ何も始まらない
- **物理的なつながり**: LANケーブル、Wi-Fi、光ファイバー
- **デジタル信号**: 0と1の電気信号に変換

**データの流れ（送信時）**:
```
[アプリケーション] "Hello!" というメッセージを送りたい
         ↓
[第7層] アプリケーションがデータを作成
         ↓
[第4層] TCPがパケットに分割、順序番号を付与
         ↓
[第3層] IPアドレスを付けて経路を決定
         ↓
[第2層] MACアドレスを付けて直接送信先を決定
         ↓
[第1層] 電気信号（0101010...）に変換してケーブルに送出
         ↓
     ケーブルを通って送信！
```

## 第5章: ネットワークの歴史

### Q6: ネットワークはいつから始まったのですか？

**A6**: ネットワークの歴史は**1960年代のARPANET**から始まりました。

**ネットワークの歴史年表**:

```
1969年: ARPANET誕生
    |      最初のネットワーク（アメリカ国防総省）
    |      4台のコンピューターを接続
    |
    |      [UCLA] ━━ [Stanford]
    |         ┃           ┃
    |      [UCSB] ━━ [Utah大学]
    |
    ↓
1970年代: TCP/IP開発
    |      現代のインターネットの基礎プロトコル
    |
    ↓
1980年代: インターネットの商用化開始
    |      大学や企業が接続
    |
    ↓
1991年: World Wide Web (WWW) 誕生
    |      Webページを見られるように
    |      (Tim Berners-Leeが開発)
    |
    ↓
1990年代: インターネットの一般普及
    |      一般家庭でもインターネット接続
    |
    ↓
2000年代: ブロードバンド時代
    |      高速インターネット（光ファイバー等）
    |
    ↓
2010年代: スマートフォン時代
    |      モバイルインターネットの普及
    |
    ↓
現在: IoT、クラウド時代
         すべてがネットワークでつながる時代
```

**初期のネットワーク図（ARPANET）**:
```
      カリフォルニア                   ユタ州
           ┌────┐                    ┌────┐
           │UCLA│                    │Utah│
           └─┬──┘                    └──┬─┘
             │                          │
    専用回線 │         専用回線          │ 専用回線
    (電話線) │        (電話線)          │ (電話線)
             │         ┌────┐           │
             └─────────┤IMP │───────────┘
                       └────┘
                    (ルーターの原型)
                         │
                         │
                     ┌───┴───┐
                  ┌──┴──┐ ┌──┴──┐
                  │UCSB │ │SRI  │
                  └─────┘ └─────┘

    IMP = Interface Message Processor
         (現代のルーターに相当)
```

**重要な進化**:
1. **少数から多数へ**: 4台 → 数億台
2. **専用線から共有へ**: 専用回線 → インターネット
3. **研究から日常へ**: 軍事・研究 → 誰でも使える

## 第6章: IPアドレス - ネットワーク上の住所

### Q7: IPアドレスって何ですか？

**A7**: IPアドレスとは、**ネットワーク上の住所**です。各コンピューターを識別するための番号です。

**住所との比較**:

| 現実世界 | ネットワーク |
|---------|-------------|
| 住所: 東京都渋谷区... | IPアドレス: 192.168.1.10 |
| 郵便番号で地域を特定 | ネットワーク部で範囲を特定 |
| 番地で家を特定 | ホスト部で機器を特定 |

**IPアドレスの構造（IPv4）**:
```
   192  .  168  .   1   .  10
   └─┬─┘  └─┬─┘  └─┬─┘  └┬┘
     │      │      │     │
  8ビット 8ビット 8ビット 8ビット
     │      │      │     │
     └──────┴──────┴─────┘
        合計32ビット


ネットワーク部  :  192.168.1.0/24
                  (このネットワークの範囲)

ホスト部        :  10
                  (この範囲内での個別の番号)
```

**例: ホームネットワーク**:
```
      インターネット
           │
           │
      ┌────┴────┐
      │ルーター  │  192.168.1.1 (ゲートウェイ)
      └────┬────┘
           │
    ───────┴───────────────
    │       │        │      │
  ┌─┴─┐  ┌─┴─┐  ┌──┴──┐  ┌─┴──┐
  │PC │  │PC │  │スマホ│  │TV  │
  └───┘  └───┘  └─────┘  └────┘
  .10    .11      .12      .13

すべて 192.168.1.x のネットワークに所属
```

**IPアドレスの種類**:

1. **プライベートIPアドレス**（家庭・社内ネットワーク）:
   ```
   192.168.0.0   〜 192.168.255.255
   172.16.0.0    〜 172.31.255.255
   10.0.0.0      〜 10.255.255.255
   ```

2. **グローバルIPアドレス**（インターネット上の住所）:
   ```
   上記以外の全てのIPアドレス
   例: 8.8.8.8 (GoogleのDNS)
   ```

**シーケンス: データが届くまで**:
```
PC (192.168.1.10) から Googleサーバー (8.8.8.8) へ

PC                 ルーター              インターネット        Google
│                    │                      │                  │
│─ パケット送信 ───→│                      │                  │
│  宛先: 8.8.8.8     │                      │                  │
│  送信元: 192.168.1.10                     │                  │
│                    │                      │                  │
│                    │─ アドレス変換 ────→│                  │
│                    │  宛先: 8.8.8.8       │                  │
│                    │  送信元: (グローバルIP)                 │
│                    │                      │                  │
│                    │                      │─ 経路選択 ─────→│
│                    │                      │                  │
│                    │                      │←─ 応答 ─────────│
│                    │                      │                  │
│                    │←─ アドレス逆変換 ──│                  │
│                    │                      │                  │
│←─ パケット受信 ───│                      │                  │
│                    │                      │                  │
```

## 第7章: ルーター - ネットワークの交差点

### Q8: ルーターって何ですか？

**A8**: ルーターとは、**異なるネットワーク同士をつなぐ機器**です。郵便局や交差点のような役割を果たします。

**郵便局との比較**:
```
手紙を送る場合:
  あなた → 地元の郵便局 → 中継郵便局 → ... → 相手の郵便局 → 相手

パケットを送る場合:
  あなたのPC → 家のルーター → ISPのルーター → ... → 相手のルーター → 相手のPC
```

**ルーターの役割**:

```
        家庭内ネットワーク              インターネット
       (192.168.1.0/24)

  [PC]     [スマホ]    [タブレット]
   │         │            │
   └────┬────┴────────────┘
        │
    ┌───┴────┐
    │ルーター │  ← 2つのネットワークに接続
    └───┬────┘
        │                    1. 内側: 192.168.1.1
        │                    2. 外側: グローバルIP
        │
   インターネット
        │
    ┌───┴────┐
    │ISPの    │
    │ルーター │
    └────────┘
```

**ルーターの3つの主な機能**:

1. **パケットの転送（ルーティング）**:
   ```
   パケット到着
        ↓
   [宛先IPアドレスを確認]
        ↓
   どのネットワークに送るか判断
        ↓
   適切なネットワークに転送
   ```

2. **NAT（Network Address Translation）**:
   ```
   プライベートIP → グローバルIP 変換

   家の中 (192.168.1.10)
        ↓
   [ルーターでNAT変換]
        ↓
   インターネット (グローバルIP: 203.0.113.5)
   ```

3. **ファイアウォール（セキュリティ）**:
   ```
   外部からの不正アクセスをブロック

   インターネット
        │
   [不審なパケット？]
        │
    ┌──┴──┐
    │YES  NO│
    │      │
   ❌    ✅ 通過許可
   ブロック
   ```

**ルーターのルーティングテーブル**:
```
宛先ネットワーク      次に送る先         インターフェース
192.168.1.0/24    ──→ 直接接続      ──→ eth0 (内側)
0.0.0.0/0         ──→ ISPルーター  ──→ eth1 (外側)
                       (デフォルトゲートウェイ)
```

## 第8章: デフォルトゲートウェイ - ネットワークの出口

### Q9: デフォルトゲートウェイって何ですか？

**A9**: デフォルトゲートウェイとは、**自分のネットワークの外に出るときに使う「出口」**です。

**建物の出口との比較**:
```
学校の中を移動する場合:
  教室A → 廊下 → 教室B  （校内なので自由に移動）

校外に出る場合:
  教室A → 廊下 → 正門 → 外  （正門 = ゲートウェイ）
```

**ネットワークでの動作**:
```
同じネットワーク内の通信:
  PC (192.168.1.10) → PC (192.168.1.11)
  直接通信できる（ゲートウェイ不要）

異なるネットワークへの通信:
  PC (192.168.1.10) → ゲートウェイ (192.168.1.1) → インターネット
                      ↑
                  必ずここを経由
```

**詳細なシーケンス図**:
```
PC (192.168.1.10) が Google (8.8.8.8) にアクセスしたい

1. PCが判断:
   「8.8.8.8 は自分のネットワーク (192.168.1.0/24) の外だ」

2. PCの動作:
   PC: 「よし、デフォルトゲートウェイ (192.168.1.1) に送ろう」

3. シーケンス:

   PC                    ゲートウェイ              インターネット
   │                         │                         │
   │─ パケット ──────────→│                         │
   │  宛先: 8.8.8.8          │                         │
   │  次の宛先: 192.168.1.1  │                         │
   │                         │                         │
   │                         │─ パケット転送 ────────→│
   │                         │  宛先: 8.8.8.8          │
   │                         │                         │
   │                         │                    [Googleサーバー]
   │                         │                    「パケット受信！」
   │                         │                         │
   │                         │←─ 応答 ────────────────│
   │                         │                         │
   │←─ 応答を転送 ──────────│                         │
   │                         │                         │
```

**設定例（Windows）**:
```
IPアドレス:           192.168.1.10
サブネットマスク:     255.255.255.0
デフォルトゲートウェイ: 192.168.1.1  ← これがルーター
```

**もしゲートウェイがないと？**:
```
PC: 「8.8.8.8 に行きたいけど、どこに送ればいいか分からない...」
  ↓
❌ インターネットにアクセスできない
```

## 第9章: サブネットマスク - ネットワークの範囲

### Q10: サブネットマスクって何ですか？

**A10**: サブネットマスクとは、**IPアドレスの「ネットワーク部」と「ホスト部」を区別するためのマスク**です。

**住所での例え**:
```
住所: 東京都渋谷区神南1-2-3

サブネットマスクで区別すると:
┌─────────────┬─────┐
│ 東京都渋谷区神南 │ 1-2-3 │
└─────────────┴─────┘
   ネットワーク部    ホスト部
   (どの地域か)     (どの家か)
```

**IPアドレスでの例**:
```
IPアドレス:      192.168.  1. 10
サブネットマスク: 255.255.255.  0
                  └───┬───┘ └┬┘
                ネットワーク部 ホスト部

意味:
- ネットワーク部: 192.168.1.0/24 という範囲
- ホスト部: その範囲内の10番目
```

**CIDR記法（スラッシュ記法）**:
```
192.168.1.0/24

/24 の意味:
- 先頭から24ビットがネットワーク部
- 残り8ビット (32-24) がホスト部

255.255.255.0 と同じ意味:
11111111.11111111.11111111.00000000
└────────24ビット────────┘└8ビット┘
```

**よく使われるサブネットマスク**:

| CIDR | サブネットマスク | ホスト数 | 用途 |
|------|-----------------|---------|------|
| /24 | 255.255.255.0 | 254台 | 家庭・小規模オフィス |
| /16 | 255.255.0.0 | 65,534台 | 大企業・Dockerデフォルト |
| /8 | 255.0.0.0 | 16,777,214台 | 超大規模ネットワーク |

**ネットワーク範囲の計算**:
```
192.168.1.0/24 の場合:

使えるIPアドレス:
- 範囲: 192.168.1.0 〜 192.168.1.255
- ネットワークアドレス: 192.168.1.0 (使用不可)
- ホストアドレス: 192.168.1.1 〜 192.168.1.254 (254台)
- ブロードキャストアドレス: 192.168.1.255 (使用不可)

通常の割り当て:
- 192.168.1.1: ルーター（ゲートウェイ）
- 192.168.1.2 〜 254: PC、スマホなどの機器
```

## 第10章: Dockerネットワークとローカルネットワーク

### Q11: Dockerのネットワークって何ですか？普通のネットワークと何が違うのですか？

**A11**: Dockerネットワークとは、**仮想的に作られたネットワーク**で、Dockerコンテナ同士を接続するためのものです。

**物理ネットワークとの違い**:

| 項目 | 物理ネットワーク | Dockerネットワーク |
|------|-----------------|-------------------|
| **実体** | 物理的なケーブル、Wi-Fi | ソフトウェアで仮想的に作成 |
| **接続機器** | PC、スマホ、サーバー | Dockerコンテナ |
| **IPアドレス範囲** | 192.168.x.x (通常) | 172.17.x.x (デフォルト) |
| **目的** | 実際の機器の接続 | 仮想環境の分離と接続 |

**全体図**:
```
┌──────────────────────────────────────────────────┐
│               物理的なPC                          │
│   (IPアドレス: 192.168.1.10)                     │
│                                                  │
│  ┌────────────────────────────────────────┐     │
│  │        Dockerネットワーク                │     │
│  │        (172.17.0.0/16)                 │     │
│  │                                        │     │
│  │  ┌──────┐  ┌──────┐  ┌──────┐        │     │
│  │  │コンテナ│  │コンテナ│  │コンテナ│        │     │
│  │  │ Web  │  │ DB   │  │Cache │        │     │
│  │  └──┬───┘  └──┬───┘  └──┬───┘        │     │
│  │     │.2       │.3       │.4           │     │
│  │     └─────┬───┴─────┬───┘            │     │
│  │           │         │                 │     │
│  │      ┌────┴─────────┴────┐           │     │
│  │      │ docker0 (ブリッジ) │           │     │
│  │      │    172.17.0.1      │           │     │
│  │      └────────┬───────────┘           │     │
│  └───────────────┼───────────────────────┘     │
│                  │                              │
│         ┌────────┴────────┐                     │
│         │  物理NIC (eth0)  │                     │
│         │  192.168.1.10    │                     │
│         └────────┬─────────┘                     │
└──────────────────┼──────────────────────────────┘
                   │
            ───────┴───────
            ローカルネットワーク
            (192.168.1.0/24)
```

**詳細なネットワーク階層**:

```
レイヤー1: 物理的なローカルネットワーク
  ├─ ルーター: 192.168.1.1
  ├─ あなたのPC: 192.168.1.10
  ├─ 他のPC: 192.168.1.11
  └─ スマホ: 192.168.1.12

レイヤー2: PCの内部 (192.168.1.10)
  ├─ 物理NIC (eth0): 192.168.1.10
  └─ Dockerブリッジ (docker0): 172.17.0.1

レイヤー3: Dockerネットワーク (172.17.0.0/16)
  ├─ Webコンテナ: 172.17.0.2
  ├─ DBコンテナ: 172.17.0.3
  └─ Cacheコンテナ: 172.17.0.4
```

### Q12: Dockerのコンテナからインターネットにアクセスできるのはなぜですか？

**A12**: **Dockerがホスト（あなたのPC）を経由して、ローカルネットワークとつないでいるから**です。

**詳細なシーケンス図**:
```
Webコンテナ (172.17.0.2) が Google (8.8.8.8) にアクセスする場合

ステップ1: コンテナからdocker0へ
  [Webコンテナ]           [docker0]
  172.17.0.2            172.17.0.1
       │                    │
       │─ パケット送信 ────→│
       │  宛先: 8.8.8.8      │
       │  送信元: 172.17.0.2  │
       │                    │

ステップ2: docker0からホストOSへ（NAT変換）
       │                [ホストOS]
       │                    │
       │                    │ NAT変換:
       │                    │ 172.17.0.2 → 192.168.1.10
       │                    │
       │─ パケット転送 ────→│
       │  宛先: 8.8.8.8      │
       │  送信元: 192.168.1.10 │
       │                    │

ステップ3: ホストOSからルーターへ
       │                [ルーター]
       │                192.168.1.1
       │                    │
       │─ パケット送信 ────→│
       │  宛先: 8.8.8.8      │
       │  送信元: 192.168.1.10 │
       │                    │

ステップ4: ルーターからインターネットへ（再度NAT変換）
       │              [インターネット]
       │                    │
       │                    │ NAT変換:
       │                    │ 192.168.1.10 → グローバルIP
       │                    │
       │─ パケット転送 ────→│
       │  宛先: 8.8.8.8      │
       │  送信元: (グローバルIP)
       │                    │
       │              [Googleサーバー]
       │                    │
       │                 「受信！」
```

**2段階のNAT変換**:
```
1段階目 (Docker内):
  172.17.0.2 (コンテナ) → 192.168.1.10 (ホストPC)

2段階目 (ルーター):
  192.168.1.10 (プライベート) → xxx.xxx.xxx.xxx (グローバル)
```

### Q13: Dockerネットワークとローカルネットワークはどうやってつながっているのですか？

**A13**: **ブリッジネットワーク**という仕組みで、仮想的な「橋渡し」をしています。

**ブリッジの仕組み**:
```
        ローカルネットワーク (192.168.1.0/24)
               │
        ───────┴─────────── (物理的なネットワーク)
               │
        ┌──────┴──────┐
        │ 物理NIC      │ 192.168.1.10
        │   (eth0)     │
        └──────┬───────┘
               │
        ┌──────┴───────┐
        │  iptables    │ ← パケット転送ルール（NAT）
        │ (Linux機能)  │
        └──────┬───────┘
               │
        ┌──────┴───────┐
        │ docker0      │ 172.17.0.1
        │ (ブリッジ)    │ ← 仮想的なスイッチ
        └──────┬───────┘
               │
    ┌──────────┼──────────┐
    │          │          │
┌───┴───┐ ┌───┴───┐ ┌───┴───┐
│veth1  │ │veth2  │ │veth3  │ ← 仮想的なケーブル
└───┬───┘ └───┬───┘ └───┬───┘
    │         │         │
┌───┴───┐ ┌───┴───┐ ┌───┴───┐
│コンテナ│ │コンテナ│ │コンテナ│
│ .2    │ │ .3    │ │ .4    │
└───────┘ └───────┘ └───────┘
   Web      DB      Cache
```

**用語解説**:

1. **docker0 (ブリッジ)**:
   - 仮想的なスイッチの役割
   - すべてのコンテナを接続
   - IPアドレス: 172.17.0.1

2. **veth (Virtual Ethernet)**:
   - 仮想的なネットワークケーブル
   - 片方がコンテナ内、もう片方がdocker0に接続

3. **iptables (NAT)**:
   - Linuxのパケット転送機能
   - アドレス変換（NAT）を実行

**パケットの流れ（詳細）**:
```
1. コンテナ内でパケット作成:
   [Webコンテナ] パケット作成
   送信元: 172.17.0.2
   宛先: 8.8.8.8

2. veth経由でdocker0へ:
   [veth1] → [docker0]

3. docker0で経路判断:
   「8.8.8.8 は外部なので、ホストOSに転送」

4. iptablesでNAT変換:
   送信元: 172.17.0.2 → 192.168.1.10
   宛先: 8.8.8.8 (変更なし)

5. 物理NIC (eth0) から送信:
   [eth0] → ローカルネットワーク → ルーター → インターネット
```

**A14**: **同じDockerネットワーク内では、直接通信できます**。docker0ブリッジを経由します。

**コンテナ間通信の例**:

```
WebコンテナがDBコンテナにアクセスする場合

┌─────────────┐              ┌─────────────┐
│ Webコンテナ  │              │ DBコンテナ   │

└──────┬──────┘              └──────┬──────┘
       │                            │
       │                            │
    [veth1]                     [veth2]

       │      ┌──────────┐          │
       └──────┤ docker0  │──────────┘

通信シーケンス:
  Web (172.17.0.2)
    │
    │─ SQLクエリ送信 ────→ docker0 ────→ DB (172.17.0.3)
    │                                     │
    │                                  [SQL実行]
    │                                     │
    │←─ 結果を返信 ────── docker0 ←─────┘
    │
```

**Dockerネットワークの種類**:

```
   ┌─────────────────────────┐

   │    172.17.0.0/16        │

   - コンテナ同士が通信可能

2. ホストネットワーク:

3. カスタムブリッジ:
   ┌──────────────┐  ┌──────────────┐
   │ Network-A    │  │ Network-B    │
   │ ┌────┐┌────┐ │  │ ┌────┐┌────┐│
   │ │Web1││DB1 │ │  │ │Web2││DB2 ││
   │ └────┘└────┘ │  │ └────┘└────┘│
   └──────────────┘  └──────────────┘
   - 複数のネットワークを作成可能
   - ネットワーク間は分離
```

## 第11章: 実践例 - LocalNetScanの仕組み

### Q15: LocalNetScanはどうやってネットワーク内の機器を見つけるのですか？

**A15**: **PINGスキャン**という技術で、ネットワーク内のすべてのIPアドレスに対して「応答確認」を行っています。

**動作の流れ**:
```
1. ネットワーク範囲を特定:
   「このPCは 192.168.1.10 だから、
    192.168.1.0/24 のネットワークに所属している」

2. すべてのIPアドレスに対してPING:
   192.168.1.1   → PING送信 → 応答あり ✅ (ルーター)
   192.168.1.2   → PING送信 → 応答なし ❌
   192.168.1.3   → PING送信 → 応答なし ❌
   ...
   192.168.1.10  → PING送信 → 応答あり ✅ (自分自身)
   192.168.1.11  → PING送信 → 応答あり ✅ (他のPC)
   ...
   192.168.1.254 → PING送信 → 応答なし ❌

3. 応答があったIPアドレスをリスト化:
   - 192.168.1.1 (ルーター)
   - 192.168.1.10 (自分)
   - 192.168.1.11 (他のPC)
```

**PINGの仕組み（ICMPプロトコル）**:
```
PC-A (192.168.1.10)              PC-B (192.168.1.11)
    │                                 │
    │─ ICMP Echo Request ───────────→│
    │  「応答してください！」           │
    │                                 │
    │                          [受信して処理]
    │                                 │
    │←─ ICMP Echo Reply ─────────────│
    │  「はい、ここにいます！」          │
    │                                 │
  「応答あり」                      「要求を受けた」
    ↓                                 ↓
 PC-Bは存在する！               正常に動作中
```

**LocalNetScanの10スレッド並列スキャン**:
```
192.168.1.0/24 を10スレッドでスキャンする場合

ネットワークを分割:
  チャンク1: 192.168.1.1-25    → スレッド1
  チャンク2: 192.168.1.26-50   → スレッド2
  チャンク3: 192.168.1.51-75   → スレッド3
  ...
  チャンク10: 192.168.1.226-254 → スレッド10

┌─────────┐ ┌─────────┐ ┌─────────┐
│スレッド1 │ │スレッド2 │ │スレッド3 │  ...
│ PING!   │ │ PING!   │ │ PING!   │
│ PING!   │ │ PING!   │ │ PING!   │
│ PING!   │ │ PING!   │ │ PING!   │
└────┬────┘ └────┬────┘ └────┬────┘
     │           │           │
     └───────────┴───────────┘
              │
         [結果を統合]
              │
    ┌─────────┴─────────┐
    │ 検出された機器:      │
    │ - 192.168.1.1      │
    │ - 192.168.1.10     │
    │ - 192.168.1.11     │
    └────────────────────┘
```

**高速化の秘密**:
```
従来（1台ずつ順番に）:
  254台 × 0.5秒 = 127秒 (約2分)

LocalNetScan（10スレッド並列 + 高速化オプション）:
  (254台 ÷ 10スレッド) × 0.2秒 = 約5秒

  約25倍高速！⚡
```

## 第12章: まとめ - ネットワークの全体像

### Q17: ネットワーク全体をまとめて説明してください

**A17**: ネットワークは**階層構造**で、それぞれの層が役割を持って連携しています。

**全体構造図**:
```
┌─────────────────────────────────────────────────┐
│           アプリケーション層                     │
│        (Web、メール、LINE等)                     │
│   「ユーザーが実際に使うアプリ」                  │
└───────────────┬─────────────────────────────────┘
                │
┌───────────────┴─────────────────────────────────┐
│          トランスポート層 (TCP/UDP)              │
│       「データを確実に届ける仕組み」              │
│    - TCPは確実（順序保証、再送）                  │
│    - UDPは高速（順序保証なし）                    │
└───────────────┬─────────────────────────────────┘
                │
┌───────────────┴─────────────────────────────────┐
│        ネットワーク層 (IP)                       │
│      「どの経路で届けるかを決定」                 │
│    - IPアドレスで宛先を特定                       │
│    - ルーターが経路を選択                         │
└───────────────┬─────────────────────────────────┘
                │
┌───────────────┴─────────────────────────────────┐
│      データリンク層 (Ethernet、Wi-Fi)            │
│    「直接つながった機器間の通信」                 │
│    - MACアドレスで識別                           │
│    - スイッチが接続を管理                         │
└───────────────┬─────────────────────────────────┘
                │
┌───────────────┴─────────────────────────────────┐
│           物理層                                 │
│      「電気信号、光信号」                         │
│    - ケーブル、Wi-Fi電波                         │
│    - 0と1のデジタル信号                           │
└─────────────────────────────────────────────────┘
```

**データ送信の全プロセス**:
```
1. アプリケーション: "Hello!" というメッセージ
         ↓
2. TCP: パケットに分割、順序番号を付与
         ↓
3. IP: 宛先IPアドレス (8.8.8.8) を追加
         ↓
4. Ethernet: MACアドレスを追加（次のルーターのMAC）
         ↓
5. 物理: 電気信号 (01001000...) に変換
         ↓
     ケーブル/Wi-Fiで送信
         ↓
    ルーターで中継（経路選択）
         ↓
    インターネットを経由
         ↓
    目的地に到着
         ↓
    逆の順序で解釈
         ↓
    "Hello!" が届く！
```

```
3. iptablesでNAT (172.17.0.2 → 192.168.1.10)
   ↓
4. 物理NIC (eth0) から送信
   ↓
5. ホームルーター経由
   ↓
6. ISPルーター経由
   ↓
7. インターネットへ
```

## 付録B: よくある質問

### Q: IPアドレスが足りなくなったらどうするの？

**A**: IPv6という新しい規格に移行しています。

```
IPv4: 32ビット（約43億個）
  例: 192.168.1.10

IPv6: 128ビット（約340澗個 - ほぼ無限）
  例: 2001:0db8:85a3:0000:0000:8a2e:0370:7334
```

---

### Q: Wi-Fiとケーブルで速度が違うのはなぜ？

**A**: 物理層の違いです。

```
有線LAN (1000BASE-T):
  最大1Gbps（理論値）
  安定した通信

Wi-Fi (Wi-Fi 6):
  最大9.6Gbps（理論値）
  実効速度は環境に依存
  障害物、距離、干渉の影響を受ける
```

---

### Q: VPNって何？

**A**: 仮想的なプライベートネットワークです。

```
通常のインターネット通信:
  あなたのPC → インターネット → サーバー
  (暗号化されていない可能性)

VPN使用時:
  あなたのPC → [暗号化トンネル] → VPNサーバー → サーバー
  (すべて暗号化)
```

## まとめ: ネットワークの本質

ネットワークとは、結局のところ:

1. **電気信号が届くこと**（物理層）
2. **その信号をパケットとして解釈すること**（データリンク層〜ネットワーク層）
3. **パケットに応答すること**（トランスポート層〜アプリケーション層）

この3つのステップが成立して初めて「つながった」と言えます。

人間のコミュニケーションと同じように:
- **音波** = 電気信号
- **言語** = プロトコル
- **会話** = パケット交換

ネットワークは、人間の知恵を技術に置き換えたものなのです。
